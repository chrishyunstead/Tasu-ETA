# Delivery ETA Prediction Lambda

> Korean version: [README_Kver.md](./README_Kver.md)

## Delivery ETA Calculation System Combining Route Optimization and Service-Time Prediction

This serverless MLOps project calculates delivery estimated time of arrival, or ETA, by combining route optimization results, OSRM-based travel time, and an item-level service-time prediction model called `tasu`.  
The calculated ETA is stored so that the TMS can query and display the latest ETA values.

> This repository is a sanitized public portfolio version. Actual operation accounts, S3 buckets, Function URLs, database table names, customer information, and tracking numbers were replaced with sample values.

## Business Problem

In delivery operations, the operations team and TMS need to know when each delivery stop is expected to be reached by the current driver.  
To calculate accurate ETA, several factors need to be considered together:

- Visit sequence generated by route optimization
- Actual travel time from depot to the first stop and from stop to stop
- Item-level service time, called `tasu`, at each delivery stop
- Recalculation for remaining stops after a delivery-completion event
- Duplicate handling for same-address or same-recipient deliveries

## End-to-End Flow

1. A dispatch or delivery-completion event occurs in the Flex App.
2. Route Optimization Lambda calculates the visit sequence and stores the result in S3.
3. ETA Calculate Lambda combines route results, OSRM travel time, and predicted `tasu` to calculate ETA.
4. Calculated results are saved to DynamoDB.
5. TMS calls the ETA Query Lambda GET API to retrieve the latest ETA.
6. ETA values are displayed on TMS map markers.

---

## ETA Calculation Logic

ETA is calculated cumulatively from the dispatch time or current time.

```text
ETA(1) = base_time + travel_time(unit в†’ stop_1) + service_time(stop_1)
ETA(n) = ETA(n-1) + travel_time(stop_n-1 в†’ stop_n) + service_time(stop_n)
```

### Processing Rules

- Initial calculation uses dispatch time as the base time.
- After delivery completion, ETA is recalculated for the remaining stops using the Lambda execution time.
- Multiple invoices with the same `address_id` are treated as one delivery stop to avoid duplicate ETA calculation.
- If sector/area information is not available in the model, a default service time is used as fallback.
- `tasu` is defined as the time required to process one item after traveling from one stop to the next.
